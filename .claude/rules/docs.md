# Knowledge Repository - ドキュメント管理規約

## 概要

このリポジトリは技術系の情報や意思決定をドキュメントとして蓄積・管理するためのナレッジベースです。

**想定ワークフロー**: 参照記事・情報 → LLM要約 → レビュー → コミット

## 1. メタドキュメント更新ルール

| 対象ファイル | 役割 | 更新タイミング | 更新必須条件 |
|-------------|------|--------------|-------------|
| **SUMMARY.md** | ナビゲーション・目次 | ドキュメント追加/削除時 | 必須 |
| **.claude/rules/docs.md** | 管理規約 | ルール変更時 | 必須 |
| **templates/*.md** | 各種テンプレート | フォーマット改善時 | 推奨 |

### 更新責任
- **SUMMARY.md**: ドキュメント追加者が必ず更新
- 新規ディレクトリ作成時は事前にルール確認・相談

## 2. 命名規則

### ディレクトリ・ファイル名
- **英語ケバブケース** を使用: `applied-information-engineer`, `network-fundamentals.md`
- **本文は日本語OK**: 可読性を優先

## 3. ファイル体系・分類

### 3.1 技術記事 (`articles/`)
**分類方式**: 技術分野別

#### 推奨ディレクトリ構造
```
articles/
├── backend/          # サーバーサイド技術
├── frontend/         # クライアントサイド技術
├── devops/          # インフラ・運用
├── data/            # データベース・分析
├── mobile/          # モバイル開発
├── security/        # セキュリティ
├── architecture/    # システム設計・アーキテクチャ
└── tools/           # 開発ツール・環境
```

### 3.2 意思決定記録 (`adr/`)
**命名**: `{3桁連番}-{決定事項}.md`
- 例: `001-database-selection.md`, `002-authentication-strategy.md`

### 3.3 応用情報技術者試験 (`applied-information-engineer/`)
**分類方式**: 試験出題分野別
- コンピューターサイエンス基礎知識を中心に分類
- 実務との関連性を重視した構造

## 4. ワークフロー定義

### 4.1 基本フロー
```mermaid
graph LR
    A[情報収集] --> B[LLM要約]
    B --> C[テンプレート適用]
    C --> D[レビュー・編集]
    D --> E[SUMMARY.md更新]
    E --> F[コミット]
```

### 4.2 各段階の詳細

#### Stage 1: 情報収集
- **対象**: 技術記事、公式ドキュメント、書籍、動画等
- **記録**: 元URL、出典情報を必ず記載

#### Stage 2: LLM要約
- **使用ツール**: Claude等のLLM
- **要約方針**:
  - 技術記事: 実務応用性を重視
  - 応用情報: 試験出題パターンと実務の関連性
  - ADR: 意思決定の背景と影響範囲

#### Stage 3: テンプレート適用
- **必須**: 該当するテンプレートファイルを使用
- **カスタマイズ**: 内容に応じてセクションの追加・削除OK

#### Stage 4: レビュー・編集
- **観点**:
  - 情報の正確性
  - 実務への適用可能性
  - 他ドキュメントとの関連性
  - 将来の参照のしやすさ

#### Stage 5: SUMMARY.md更新
- **必須事項**: パスと要約の一行説明を追加
- **分類**: ディレクトリ構造に沿った整理

#### Stage 6: コミット
- **コミットメッセージ**:
  - `add: [分野] - [タイトル]` (新規追加)
  - `update: [ファイル名] - [更新内容]` (更新)

### 4.3 品質チェックポイント

#### 新規ドキュメント作成時
- [ ] 適切なテンプレートを使用
- [ ] メタデータ（タグ、重要度等）を記載
- [ ] 関連する既存ドキュメントへのリンク
- [ ] SUMMARY.mdの更新

#### 定期メンテナンス
- [ ] 古い情報の更新確認（四半期毎）
- [ ] リンク切れのチェック
- [ ] タグの統一・整理

## 5. 相関関係・影響範囲

### 5.1 ドキュメント間の関連性

| 編集対象 | 確認が必要なファイル | 確認項目 |
|----------|-------------------|----------|
| **articles/** | 関連するadr/, applied-information-engineer/ | 技術選択の背景、学習内容との整合性 |
| **adr/** | 関連するarticles/, 実装に関わるapplied-information-engineer/ | 技術的根拠、実装可能性 |
| **applied-information-engineer/** | 関連するarticles/, adr/ | 実務での応用例、技術選択への影響 |

### 5.2 タグ・分類の統一

#### 推奨タグ体系
**技術分野**: `backend`, `frontend`, `devops`, `mobile`, `data`, `security`
**重要度**: `critical`, `important`, `reference`
**学習ステータス**: `learning`, `understood`, `review-needed` (応用情報用)
**プロジェクト**: 具体的なプロジェクト名

### 5.3 横断的検索・参照

#### 情報の発見性向上
- **タグの活用**: 複数観点からの分類
- **相互リンク**: 関連ドキュメントの明記
- **SUMMARY.md**: 全体俯瞰とナビゲーション

## 6. 運用ガイドライン

### 6.1 ドキュメント作成の判断基準

#### 作成すべきもの
- **技術記事**: 実務で活用できる具体的な技術情報
- **ADR**: チーム・プロジェクトの重要な技術選択
- **応用情報**: 試験対策として体系化すべき基礎知識

#### 作成不要なもの
- 一時的なメモやスクラップ
- 個人的すぎる内容
- 著作権的に問題のある全文コピー

### 6.2 情報の鮮度管理
- **更新頻度**: 最低年1回の見直し
- **廃止判断**: 技術が陳腐化、業務で使用しなくなった場合
- **アーカイブ**: 削除ではなく、廃止日付とステータスを記載
